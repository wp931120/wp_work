## 这是个啥：什么是 Claude Skills？

先给个一句话定义：

**Claude Skills（技能包）** 是一种结构化知识包，让你可以一次性教会 Claude 你的工作流程，然后在后续任务中一致性地应用这些知识。

通俗点说，技能包就像是给 Claude 发的 **操作手册 + 工具箱**。

它有几个关键特征：

1. **可复用性**：教一次，反复使用
2. **结构化**：包含指令、代码模板、资源文件
3. **一致性**：确保 Claude 每次都按相同方式执行任务
4. **可移植**：基于文件系统，易于编辑、版本控制、共享

**与 RAG、Few-shot 的区别：**

| 增强方式 | 内容类型 | 作用机制 |
|---------|---------|---------|
| **技能包 (Skills)** | 程序性知识（SOP、工作流） | 指导"怎么做" |
| **RAG** | 事实性检索 | 提供背景信息 |
| **Few-shot** | 示例对 | 模式匹配 |
| **系统提示** | 通用指令 | 行为约束 |

## 核心发现：SkillsBench 的 7 个关键结论

在深入技能包开发之前，笔者想先分享一个重要的基准测试结果。

论文团队构建了一个叫 **SkillsBench** 的基准测试框架，做了大规模实证研究：

- **86 个任务**，覆盖 **11 个领域**
- **7 种 Agent-模型组合**
- **7308 次测试轨迹**

### 发现 1：技能包提升显著，但差异很大

**平均提升 16.2 个百分点**，但不同配置差异很大：

- 最低：+13.6pp
- 最高：+23.3pp

这说明技能包的效果 **强烈依赖于具体的 Agent-模型组合**，不是一刀切的。

### 发现 2：Gemini CLI + Gemini 3 Flash 表现最佳

在所有组合中，Gemini CLI 配合 Gemini 3 Flash 模型达到了最高性能。

### 发现 3：自生成技能包几乎没有帮助 ⚠️

这个发现很反直觉：

**模型自己生成的技能包，平均提升几乎为零，有些甚至是负面的。**

这说明：**模型无法可靠地创作出它自己能从中受益的程序性知识。**

### 发现 4：2-3 个聚焦的技能包效果最好

大而全的技能包文档往往效果不佳。

**最佳实践是使用 2-3 个聚焦的技能包模块**，每个模块解决一个具体问题。

### 发现 5：小模型 + 技能包 ≈ 大模型

一个有趣的发现是：

**较小的模型配合技能包，可以达到与大模型相当的性能。**

这意味着技能包可以显著降低使用成本。

### 发现 6：领域差异巨大

不同领域的技能包效果差异显著：

- 医疗保健：+51.9pp（提升最大）
- 数据分析：+25.3pp
- 数学：+18.7pp
- 软件工程：+4.5pp（提升最小）

### 发现 7：19% 的任务用了技能包反而更差

**这不是万能药。**

16/84 的任务出现负收益，说明技能包设计不当可能适得其反。

## 核心原理：如何构建有效的技能包

根据 Anthropic 官方指南，构建一个有效的技能包需要遵循以下原则：

### 1. 明确技能包的目标工作流

**不要试图解决所有问题。**

一个好的技能包应该：

- 聚焦于 **2-3 个核心工作流**
- 针对特定场景提供明确的步骤指导
- 包含可执行的代码模板和检查清单

**示例：代码审查技能包**

```markdown
## 代码审查流程

1. **理解需求**：首先询问用户代码的功能目标
2. **静态分析**：检查代码结构、命名规范、注释完整性
3. **安全检查**：识别潜在的安全漏洞（SQL 注入、XSS 等）
4. **性能建议**：指出可能的性能瓶颈
5. **输出报告**：按优先级列出问题清单
```

### 2. 结构化技能包组件

一个完整的技能包应包含：

```
skill-name/
├── SKILL.md          # 主指令文件（必需）
├── resources/        # 资源文件（可选）
│   ├── templates/    # 代码模板
│   ├── examples/     # 示例代码
│   └── configs/      # 配置文件
└── tests/            # 测试用例（推荐）
```

**SKILL.md 核心结构：**

```markdown
# 技能包名称

## 适用场景
- 场景 1
- 场景 2

## 执行流程
1. 第一步：...
2. 第二步：...

## 输出规范
- 格式要求
- 质量标准

## 注意事项
- ⚠️ 常见陷阱
- 边界条件
```

### 3. 使用确定性验证器

**这是技能包设计的关键纪律。**

每个技能包应该包含验证逻辑，确保 Claude 的输出符合预期：

```python
# 验证器示例
def verify_code_review(output):
    required_sections = ["安全问题", "性能建议", "代码规范"]
    return all(section in output for section in required_sections)
```

### 4. 迭代测试和优化

**不要指望一次就完美。**

Anthropic 建议的测试流程：

1. **初步测试**：用 3-5 个典型任务测试技能包
2. **收集反馈**：记录失败案例和边界情况
3. **优化指令**：根据失败模式调整技能包内容
4. **回归测试**：确保优化不影响已有功能

## 代码实战：构建你的第一个技能包

下面笔者带你从零开始构建一个实用的技能包。

### 第 1 步：选择目标工作流

假设我们要构建一个 **GitHub PR 描述生成器** 技能包。

**目标**：让 Claude 根据代码变更自动生成高质量的 PR 描述。

### 第 2 步：创建 SKILL.md 文件

```markdown
# GitHub PR 描述生成器

## 适用场景
- 为代码变更生成清晰的 PR 描述
- 确保 PR 包含所有必要信息
- 遵循团队的 PR 模板规范

## 输入要求
用户提供：
1. Git diff 或变更摘要
2. 相关 Issue 编号（如有）
3. 测试说明（如有）

## 输出模板

```markdown
## 变更概述
[用 2-3 句话说明这次 PR 的目的]

## 主要改动
- [ ] 功能 1
- [ ] 功能 2

## 测试计划
[说明如何验证这些改动]

## 相关 Issue
Fixes #XXX

## 检查清单
- [ ] 代码已通过本地测试
- [ ] 更新了相关文档
- [ ] 添加了单元测试
```

## 质量标准
- 概述必须清晰简洁（不超过 3 句话）
- 主要改动必须用动词开头
- 测试计划必须可执行

## 注意事项
- ⚠️ 不要猜测未提供的信息
- ⚠️ 如果变更涉及敏感操作，必须提醒用户
```

### 第 3 步：添加资源文件（可选）

创建 `resources/templates/pr_template.md`：

```markdown
## 变更概述
{{summary}}

## 主要改动
{{changes}}

## 测试计划
{{testing}}

## 相关 Issue
{{issues}}

## 检查清单
{{checklist}}
```

### 第 4 步：测试技能包

用以下测试用例验证：

```bash
# 测试用例 1：简单功能修复
git diff feature/login-fix | claude --skill pr-describer

# 测试用例 2：大型重构
git diff feature/major-refactor | claude --skill pr-describer

# 测试用例 3：紧急热修复
git diff hotfix/critical-bug | claude --skill pr-describer
```

### 第 5 步：收集反馈并优化

根据测试结果，你可能需要：

- 增加对特定框架的支持（如 Django、React）
- 添加更多检查清单项目
- 优化输出格式以适应团队规范

## 深度思考：技能包的未来与局限

### 技能包确实是个很强大的工具

但笔者必须坦诚地指出几个关键点：

### 1. 局限性

**技能包不是万能药：**

- **依赖设计质量**：糟糕的技能包可能比没有更差
- **无法替代思考**：复杂决策仍需人工判断
- **维护成本**：工作流变化时需要更新技能包
- **模型依赖**：不同模型对同一技能包的响应可能不同

### 2. 横向对比判断

**什么时候用技能包，什么时候不用？**

| 场景 | 建议 | 理由 |
|------|------|------|
| 重复性高、流程固定的任务 | ✅ 强烈推荐 | ROI 最高 |
| 需要创造性、灵活性的任务 | ❌ 不推荐 | 限制发挥 |
| 团队协作、需要标准化 | ✅ 推荐 | 确保一致性 |
| 个人探索性使用 | ⚠️ 可选 | 可能限制灵活性 |

### 3. 笔者的判断

笔者认为，技能包的核心价值在于：

**将隐性知识显性化，将个人经验组织化。**

这不仅仅是技术问题，更是知识管理的范式转变。

从长期来看，**技能包可能成为企业 AI 战略的核心资产**：

- 资深员工的经验可以封装成技能包传承
- 最佳实践可以标准化并大规模复用
- 新员工可以通过技能包快速上手

### 4. 跨领域类比

**技能包就像编程中的函数：**

- 把常用逻辑封装起来，避免重复代码
- 通过参数化适应不同场景
- 可以组合使用构建更复杂的流程

**但技能包也像双刃剑：**

- 用得好：效率倍增，质量一致
- 用得差：束缚创造力，产生依赖

这暗合了一个朴素的道理：**工具的价值取决于使用者的智慧。**

---

## 结语：大道至简

不得不感叹一句：**最好的技术往往是最简单的。**

技能包的本质不是复杂的工程，而是：

1. **想清楚**：明确你要解决什么问题
2. **写下来**：把流程清晰地描述出来
3. **用起来**：在实践中不断迭代优化

从 SkillsBench 的实证研究到 Anthropic 的官方指南，核心建议都是一致的：

> **Focused Skills with 2–3 modules outperform comprehensive documentation.**
>
> **2-3 个聚焦的技能包模块，胜过包罗万象的综合文档。**

希望读者能够有所收获。

如果你正在考虑为团队引入技能包体系，笔者的建议是：

**从一个具体的、高价值的工作流开始，小步快跑，快速迭代。**

数据决定了结果的上限，技能包只是帮助你接近这个上限。

真正的核心竞争力，始终是你对工作流的深刻理解和持续优化。
