## 开头：AI Agent 的"技能包"热潮背后，Anthropic 给出了官方答案

最近 AI Agent 技能包（Skills）的概念火遍开发者社区。

Anthropic 的 Claude Code、Google 的 Gemini CLI、OpenAI 的 Codex CLI，这些终端里的 AI 助手都在支持某种形式的"技能包"。

社区仓库里已经涌现出成千上万个用户贡献的技能，涵盖软件工程、数据分析、企业工作流等各种场景。

但很多开发者心里都有个疑问：**这些技能包，到底该怎么用才有效？**

**Anthropic 官方给出了答案。**

1 月 29 日，Anthropic 发布了一篇完整的技能包开发指南，详细介绍了如何构建、测试和分发 Claude 技能包。这篇指南不仅回答了"技能包有没有用"的问题，还给出了大量实战经验和最佳实践。

这篇文章，笔者结合 SkillsBench 基准测试的实证研究，带你拆解 Anthropic 官方的技能包开发方法论。

## 这是个啥：什么是 Claude Skills？

先给个一句话定义：

**Claude Skills（技能包）** 是一种结构化知识包，让你可以一次性教会 Claude 你的工作流程，然后在后续任务中一致性地应用这些知识。

通俗点说，技能包就像是给 Claude 发的 **操作手册 + 工具箱**。

它有几个关键特征：

1. **可复用性**：教一次，反复使用
2. **结构化**：包含指令、代码模板、资源文件
3. **一致性**：确保 Claude 每次都按相同方式执行任务
4. **可移植**：基于文件系统，易于编辑、版本控制、共享

**与 RAG、Few-shot 的区别：**

| 增强方式 | 内容类型 | 作用机制 |
|---------|---------|---------|
| **技能包 (Skills)** | 程序性知识（SOP、工作流） | 指导"怎么做" |
| **RAG** | 事实性检索 | 提供背景信息 |
| **Few-shot** | 示例对 | 模式匹配 |
| **系统提示** | 通用指令 | 行为约束 |

## 核心发现：SkillsBench 的 7 个关键结论

在深入技能包开发之前，笔者想先分享一个重要的基准测试结果。

论文团队构建了一个叫 **SkillsBench** 的基准测试框架，做了大规模实证研究：

- **86 个任务**，覆盖 **11 个领域**
- **7 种 Agent-模型组合**
- **7308 次测试轨迹**

### 发现 1：技能包提升显著，但差异很大

**平均提升 16.2 个百分点**，但不同配置差异很大：

- 最低：+13.6pp
- 最高：+23.3pp

这说明技能包的效果 **强烈依赖于具体的 Agent-模型组合**，不是一刀切的。

### 发现 2：Gemini CLI + Gemini 3 Flash 表现最佳

在所有组合中，Gemini CLI 配合 Gemini 3 Flash 模型达到了最高性能。

### 发现 3：自生成技能包几乎没有帮助 ⚠️

这个发现很反直觉：

**模型自己生成的技能包，平均提升几乎为零，有些甚至是负面的。**

这说明：**模型无法可靠地创作出它自己能从中受益的程序性知识。**

### 发现 4：2-3 个聚焦的技能包效果最好

大而全的技能包文档往往效果不佳。

**最佳实践是使用 2-3 个聚焦的技能包模块**，每个模块解决一个具体问题。

### 发现 5：小模型 + 技能包 ≈ 大模型

一个有趣的发现是：

**较小的模型配合技能包，可以达到与大模型相当的性能。**

这意味着技能包可以显著降低使用成本。

### 发现 6：领域差异巨大

不同领域的技能包效果差异显著：

- 医疗保健：+51.9pp（提升最大）
- 数据分析：+25.3pp
- 数学：+18.7pp
- 软件工程：+4.5pp（提升最小）

### 发现 7：19% 的任务用了技能包反而更差

**这不是万能药。**

16/84 的任务出现负收益，说明技能包需要精心设计，不是随便写写就管用。

## Anthropic 官方的技能包开发方法论

接下来是重头戏，Anthropic 官方指南的核心内容。

### 技能包的三层架构：渐进式披露

Anthropic 提出了一个关键设计原则：**渐进式披露（Progressive Disclosure）**。

技能包采用三层结构：

![图2](claude-skills-guide_images/diagram_2_mermaid.png)

**第一层（YAML 元数据）：**
- 始终加载到 Claude 的系统提示中
- 提供足够的信息让 Claude 知道何时使用技能包
- 不会占用太多上下文

**第二层（SKILL.md 主体）：**
- 当 Claude 认为技能包相关时才加载
- 包含完整的指令和指导
- 详细的工作流程说明

**第三层（引用文件）：**
-  bundled 在技能包目录中的额外文件
- Claude 根据需要选择性地读取
- 包括 API 文档、示例代码等

这种设计的好处是 **最小化 token 使用，同时保持专业知识**。

### 技能包的三种使用场景

Anthropic 观察到三种常见的技能包使用场景：

#### 场景 1：文档与资产创建

**用途：** 创建一致、高质量的输出，包括文档、演示文稿、应用、设计、代码等。

**官方示例：** [frontend-design skill](https://github.com/anthropics/skills/tree/main/skills/frontend-design)

> "创建独特的、生产级的前端界面，具有高设计质量。当构建 Web 组件、页面、工件、海报或应用时使用。"

**关键技术：**
- 嵌入式风格指南和品牌标准
- 模板结构确保输出一致
- 最终化前的质量检查清单
- 无需外部工具 - 使用 Claude 的内置能力

#### 场景 2：工作流自动化

**用途：** 多步骤流程，受益于一致的方法论，包括跨多个 MCP 服务器的协调。

**官方示例：** [skill-creator skill](https://github.com/anthropics/skills/blob/main/skills/skill-creator/SKILL.md)

> "创建新技能的交互式指南。引导用户完成用例定义、元数据生成、指令编写和验证。"

**关键技术：**
- 带验证关卡的逐步工作流程
- 常见结构的模板
- 内置审查和改进建议
- 迭代优化循环

#### 场景 3：MCP 增强

**用途：** 工作流指导，增强 MCP 服务器提供的工具访问。

**官方示例：** [sentry-code-review skill（来自 Sentry）](https://github.com/getsentry/sentry-for-claude/tree/main/skills)

> "使用 Sentry 的错误监控数据，通过 MCP 自动分析和修复 GitHub Pull Request 中检测到的错误。"

**关键技术：**
- 按顺序协调多个 MCP 调用
- 嵌入领域专业知识
- 提供用户否则需要指定的上下文
- 常见 MCP 问题的错误处理

### 技能包的文件结构

官方推荐的技能包结构：

```
your-skill-name/
├── SKILL.md                  # 必需 - 主技能文件
├── scripts/                  # 可选 - 可执行代码
│    ├── process_data.py      # 示例
│    └── validate.sh          # 示例
├── references/               # 可选 - 文档
│    ├── api-guide.md         # 示例
│    └── examples/            # 示例
└── assets/                   # 可选 - 模板等
     └── report-template.md   # 示例
```

**关键规则：**

- **SKILL.md 命名：** 必须正好是 `SKILL.md`（区分大小写）
- **文件夹命名：** 使用短横线分隔的小写（kebab-case），如 `notion-project-setup`
- **不要 README.md：** 技能包文件夹内不要包含 `README.md`

### YAML 元数据：最重要的部分

YAML 元数据是 Claude 决定是否加载技能包的方式。这部分最关键。

**最小必需格式：**

```yaml
---
name: your-skill-name
description: 做什么用。当用户要求 [具体短语] 时使用。
---
```

**字段要求：**

**name（必需）：**
- 仅使用短横线分隔的小写（kebab-case）
- 不能有空格或大写字母
- 应与文件夹名称匹配

**description（必需）：**
- **必须**包含两者：
  - 技能包做什么
  - 何时使用它（触发条件）
- 少于 1024 个字符
- 没有 XML 标签（`<` 或 `>`）
- 包括用户可能说的具体任务
- 如果相关，提到文件类型

**好的 description 示例：**

```yaml
# 好 - 具体且可操作
description: 分析 Figma 设计文件并生成交接文档。当用户上传 .fig 文件、
  询问"设计规格"、"组件文档"或"设计到代码交接"时使用。

# 好 - 包括触发短语
description: 管理 Linear 项目工作流，包括冲刺规划、任务创建和状态跟踪。
  当用户提到"冲刺"、"Linear 任务"、"项目规划"或要求"创建工单"时使用。

# 好 - 清晰的价值主张
description: PayFlow 的端到端客户入职工作流。处理账户创建、支付设置
  和订阅管理。当用户说"新客户入职"、"设置订阅"或"创建 PayFlow 账户"时使用。
```

**不好的 description 示例：**

```yaml
# 太模糊
description: 帮助处理项目。

# 缺少触发条件
description: 创建复杂的多页文档系统。

# 太技术化，没有用户触发
description: 实现具有层次关系的项目实体模型。
```

## 代码实战：构建一个 GitHub PR 描述生成器

理论说了这么多，来点实际的。

笔者结合官方指南，写一个实用的技能包：**GitHub PR 描述生成器**。

### 用例定义

```
用例：GitHub PR 描述生成
触发：用户说"帮我写 PR 描述"或"生成 PR 说明"
步骤：
1. 读取 git diff 了解变更内容
2. 分析变更类型（功能/修复/重构）
3. 生成结构化的 PR 描述
4. 输出 Markdown 格式
结果：完整的 PR 描述，包括变更摘要、测试计划、检查清单
```

### SKILL.md 文件

```markdown
---
name: github-pr-description
description: 自动生成 GitHub Pull Request 描述。当用户要求"写 PR 描述"、
  "生成 PR 说明"、"create PR description"或上传 git diff 时使用。
---

# GitHub PR 描述生成器

## 指令

### 步骤 1：分析变更内容

读取用户提供的 git diff 或变更文件列表，识别：
- 变更类型（新功能/修复/重构/文档）
- 影响范围（核心模块/边缘功能）
- 潜在风险（破坏性变更/向后兼容）

### 步骤 2：生成 PR 描述结构

按照以下模板生成：

```markdown
## 摘要
一句话描述这个 PR 做什么。

## 变更类型
- [ ] 新功能
- [ ] Bug 修复
- [ ] 重构
- [ ] 文档更新

## 测试计划
- [ ] 单元测试已添加
- [ ] 手动测试已完成
- [ ] 回归测试已通过

## 检查清单
- [ ] 代码符合团队规范
- [ ] 已更新相关文档
- [ ] 无破坏性变更
```

### 步骤 3：添加具体变更详情

根据 diff 内容，列出关键变更：
- 新增的文件和功能
- 修复的问题
- 重构的模块

## 示例

### 示例 1：新功能 PR

用户说："帮我写个 PR 描述，新增了用户登录功能"

输出：
```markdown
## 摘要
新增用户登录功能，支持邮箱密码登录和第三方 OAuth。

## 变更类型
- [x] 新功能
- [ ] Bug 修复
- [ ] 重构
- [ ] 文档更新

## 测试计划
- [x] 单元测试已添加
- [x] 手动测试已完成
- [ ] 回归测试已通过

## 检查清单
- [x] 代码符合团队规范
- [x] 已更新相关文档
- [ ] 无破坏性变更

## 关键变更
- 新增 `auth/login.ts` 处理登录逻辑
- 新增 `components/LoginForm.tsx` 登录表单组件
- 更新 `routes.ts` 添加登录路由
```

## 故障排除

### 问题：diff 内容太多
**原因：** 变更文件超过 50 个
**解决：** 建议用户按模块分组提交，或只关注核心变更

### 问题：无法确定变更类型
**原因：** diff 信息不足
**解决：** 询问用户这个 PR 的主要目的是什么
```

## 技能包的测试与迭代

Anthropic 强调，技能包需要测试和迭代。

### 推荐的测试方法

#### 1. 触发测试

**目标：** 确保技能包在正确的时候加载。

**测试用例：**

```
应该触发：
- "帮我写个 PR 描述"
- "生成 GitHub PR 说明"
- "create PR description for this change"

不应该触发：
- "今天的天气怎么样"
- "帮我写 Python 代码"
- "创建 Excel 表格"（除非技能包处理表格）
```

#### 2. 功能测试

**目标：** 验证技能包产生正确的输出。

**测试用例：**
- 生成有效的输出
- API 调用成功
- 错误处理正确

### 迭代策略

Anthropic 建议的迭代策略：

**先迭代单个任务，再扩展：**

> 最有效的技能包创建者会先迭代单个困难任务，直到 Claude 成功，然后将成功的方法提取到技能包中。这利用了 Claude 的上下文学习能力，比广泛测试提供更快的反馈。一旦有了工作的基础，再扩展到多个测试用例以确保覆盖。

## 技能包 + MCP：1+1>2 的组合

Anthropic 指南中特别提到了技能包与 MCP（Model Context Protocol）的结合。

### 厨房类比

**MCP** 提供专业厨房：访问工具、食材和设备。

**技能包** 提供食谱：如何创建有价值东西的逐步说明。

两者结合，让用户能够完成复杂任务，而无需自己摸索每一步。

### 为什么技能包对 MCP 用户很重要

**没有技能包：**
- 用户连接了 MCP 但不知道下一步做什么
- 支持工单询问"如何用你的集成做 X"
- 每次对话都从零开始
- 结果不一致，因为用户每次提示不同
- 用户责怪连接器，而真正的问题是工作流指导

**有技能包：**
- 预构建的工作流在需要时自动激活
- 一致、可靠的工具使用
- 最佳实践嵌入每次交互
- 降低学习曲线

## 结语：技能包的未来

Anthropic 的官方指南给出了一个清晰的信号：

**技能包不是噱头，而是 AI Agent 发展的必然方向。**

就像软件开发的演进：

- 汇编语言 → 高级语言 → 框架/库 → 低代码/无代码
- AI 交互也在经历：原始提示 → 结构化提示 → 技能包 → 自主 Agent

技能包的核心价值在于 **将隐性知识显性化**。

每个开发者、每个团队都有自己的工作方法和最佳实践。技能包让这些东西可以传承、可以复用、可以规模化。

**笔者的判断：**

1. **技能包会成为 AI Agent 的标配**，就像现代编程离不开框架和库
2. **技能包市场会形成**，优秀的技能包会像开源项目一样获得关注和采用
3. **技能包工程化是下一步**，会出现技能包的测试框架、性能分析工具、调试工具

**局限性：**

- 技能包不是万能药，需要精心设计
- 模型自己生成的技能包效果不佳，需要人类专家参与
- 不同领域效果差异大，需要针对性优化

**展望：**

技能包确实是个很有潜力的东西，这暗合了软件工程的核心思想：**复用优于重复，抽象优于具体**。

不得不感叹一句：AI 时代，知识的传承方式正在发生根本性变化。

希望读者能够有所收获，在自己的工作流中尝试技能包，找到适合自己的最佳实践。
