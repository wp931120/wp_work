# 12 节课手搓一个 AI Agent：从"人工智障"到"靠谱队友"的进化之路

> 模型就是智能体。我们的工作就是给它工具，然后让开。

---

## 一、你是不是也被"AI Agent"这个词唬住过？

说实话，我第一次听到"Agent"这个词时，脑子里浮现的是《黑客帝国》里的特工史密斯，或者至少是个能独立思考、自主决策的高级 AI。

直到我遇到了这个项目 —— **[learn-claude-code](https://github.com/shareAI-lab/learn-claude-code)**。

它用 12 节课，从零开始手搓一个 AI Agent，每节课只加一个机制，每句话都说人话。学完我才发现：**原来 Agent 的核心代码，不到 50 行。**

更颠覆的是那句格言：

> **"模型就是智能体。我们的工作就是给它工具，然后让开。"**

这篇文章，我想带你走一遍这 12 节课的进化之路。不是为了让你成为专家，而是想告诉你：**Agent 没你想的那么神秘，它就是个"会调用工具的循环"。**

## 二、核心模式：一个循环，搞定一切

先上代码。这是整个项目的核心，也是所有 AI Agent 的心脏：

```python
def agent_loop(messages):
    while True:
        response = client.messages.create(
            model=MODEL, system=SYSTEM,
            messages=messages, tools=TOOLS,
        )
        messages.append({"role": "assistant", "content": response.content})

        if response.stop_reason != "tool_use":
            return

        results = []
        for block in response.content:
            if block.type == "tool_use":
                output = TOOL_HANDLERS[block.name](**block.input)
                results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,
                    "content": output,
                })
        messages.append({"role": "user", "content": results})
```

看懂了吗？逻辑简单到离谱：

1. **把消息发给 LLM**
2. **如果它说"我要用工具"** → 执行工具，把结果塞回去
3. **如果它不说用工具** → 返回结果，结束

就这么简单。但问题来了：**这么简单东西，凭什么叫"智能体"？**

答案藏在 12 节课里。每节课加一个机制，像搭积木一样，把这个"人工智障"养成"靠谱队友"。

## 三、第一阶段：循环与工具（s01-s02）

### s01：一个循环就够了

第一节课的格言是：

> **"One loop & Bash is all you need"**

翻译成人话：**一个工具 + 一个循环 = 一个智能体**

这时候的 Agent 像个刚出生的婴儿，只会一件事：收到命令，调 Bash 执行，返回结果。

```
User --> messages[] --> LLM --> response
                                  |
                        stop_reason == "tool_use"?
                       /                          \
                     yes                           no
                      |                             |
                execute tools                    return text
                append results
                loop back -----------------> messages[]
```

这个图看起来简单，但它是所有 AI Agent 的 DNA。后面所有的复杂机制，都是在这个循环上"打补丁"。

### s02：加一个工具，只加一个 handler

第二节课教你扩展能力：

> **"加一个工具，只加一个 handler"**

想加个文件读写工具？写个 handler，注册进 dispatch map 就行。主循环？不用动。

这就像给手机装 App —— 系统内核不用改，装个新应用就能干新活。

**但这时候的 Agent 有个致命问题：它没有计划。**

你让它"写个网站"，它可能直接开始写 HTML，写到一半发现忘了数据库设计，然后推倒重来。

## 四、第二阶段：规划与知识（s03-s06）

### s03：没有计划的 agent 走哪算哪

第三节课上了第一个关键机制：**TodoWrite**

> **"没有计划的 agent 走哪算哪"**

这一步太关键了。就像你接了个大项目，第一件事不是动手干，而是列个清单：

1. 需求分析
2. 技术方案
3. 编码实现
4. 测试验证

Agent 也一样。有了计划，完成率直接翻倍。

**这里有个细节：** 很多人以为 Agent 的"计划"是 LLM 自己想出来的。其实不是 —— 是你在 system prompt 里告诉它："你有个工具叫 TodoWrite，干活前先列计划"。

这就是**机制设计**的力量。

### s04：大任务拆小，每个小任务干净的上下文

第四节课引入**子智能体（Subagents）**：

> **"大任务拆小，每个小任务干净的上下文"**

想象一下这个场景：你让 Agent 开发一个全栈应用。它要同时处理前端、后端、数据库、部署...

如果不拆分会怎样？messages[] 会越来越长，上下文污染，Agent 开始胡言乱语。

子智能体的做法是：**每个子任务启动一个独立的 Agent，用独立的 messages[]，互不干扰。**

主 Agent 像项目经理，子 Agent 像执行员工。项目经理只关心"做什么"，不关心"怎么做"。

### s05：用到什么知识，临时加载什么知识

第五节课教**Skills 机制**：

> **"用到什么知识，临时加载什么知识"**

这个设计非常反直觉。大多数人的第一反应是：把知识都塞进 system prompt 啊！

但这样做的代价是：每次调用 LLM 都要带着这些知识，token 爆炸，速度变慢，成本飙升。

Skills 的做法更聪明：**平时不加载，用到时再读取 SKILL.md 文件，通过 tool_result 注入。**

就像你不会把整本 Python 教程背下来，而是遇到问题再查文档。

### s06：上下文总会满，要有办法腾地方

第六节课是**Context Compact**：

> **"上下文总会满，要有办法腾地方"**

这是所有长对话 Agent 的终极问题。LLM 的 context window 是有限的，对话长了必然爆掉。

解决方案是**三层压缩策略**：

1. **最近对话**：保留最新的 N 轮对话（热数据）
2. **摘要层**：把早期对话压缩成摘要（温数据）
3. **归档层**：把完整历史存到磁盘，需要时再加载（冷数据）

这个设计像极了操作系统的内存管理 —— 你想想，是不是同一个思路？

## 五、第三阶段：持久化（s07-s08）

### s07：大目标要拆成小任务，排好序，记在磁盘上

第七节课上了**任务系统**：

> **"大目标要拆成小任务，排好序，记在磁盘上"**

之前的 TodoWrite 是内存里的，Agent 重启就没了。s07 把任务持久化到磁盘，变成文件。

更重要的是：**任务之间可以有依赖关系**。

```
任务 A (数据库设计) → 任务 B (API 开发) → 任务 C (前端对接)
```

这就形成了一个**任务图（Task Graph）**。Agent 可以自动判断哪些任务可以并行，哪些必须串行。

**这个设计是为后面的多 Agent 协作打基础。** 你想想，如果任务不持久化，怎么分给不同的 Agent？

### s08：慢操作丢后台，Agent 继续想下一步

第八节课引入**后台任务**：

> **"慢操作丢后台，Agent 继续想下一步"**

想象这个场景：你让 Agent 跑一个耗时 10 分钟的测试。

如果没有后台机制，Agent 就只能傻等 10 分钟，什么都干不了。

s08 的做法是：**启动一个守护线程跑命令，Agent 继续处理其他任务。完成后注入通知。**

这就像你下了个大型文件，不用盯着进度条，可以继续干别的。

## 六、第四阶段：团队（s09-s12）

从这里开始，事情变得有趣了。

### s09：任务太大一个人干不完，要能分给队友

第九节课引入**Agent 团队**：

> **"任务太大一个人干不完，要能分给队友"**

每个队友是持久化的，有独立的"邮箱"（JSONL 文件）。主 Agent 发消息到邮箱，队友醒来处理。

**这里的细节很妙：** 队友不是"调用"就完事，它是持久存在的。这意味着它可以有自己的记忆、状态、甚至"性格"。

### s10：队友之间要有统一的沟通规矩

第十节课定义**团队协议**：

> **"队友之间要有统一的沟通规矩"**

想象一个团队，每个人说话方式都不一样：有人用邮件，有人用微信，有人用飞书... 协作效率能高吗？

s10 定义了统一的 **request-response 模式**，所有沟通都走这个协议。

更关键的是**状态机（FSM）**：队友收到请求后，可以"接受"、"拒绝"、"协商"，每种状态都有明确的流转规则。

### s11：队友自己看看板，有活就认领

第十一节课是**自治智能体**：

> **"队友自己看看板，有活就认领"**

这是从"被动执行"到"主动工作"的转折点。

之前的队友都是等主 Agent 分配任务。s11 的队友会**定期轮询任务板，看到适合自己的任务就自动认领**。

这像极了成熟的工程团队 —— 不需要领导逐个分配，大家自己看 Jira 认领。

### s12：各干各的目录，互不干扰

第十二节课是**Worktree 隔离**：

> **"各干各的目录，互不干扰"**

这是最后一块拼图。每个任务有自己的工作目录（worktree），代码、文件、环境完全隔离。

**为什么这个重要？** 想象两个队友同时改一个项目，一个改前端，一个改后端。如果不隔离，git 冲突、文件覆盖、环境冲突... 直接乱套。

s12 的做法是：**任务 ID 绑定 worktree ID，每个任务在自己的沙盒里工作。**

## 七、学完之后呢？从理解到落地

12 节课走完，你已经理解了 Agent 的完整架构。但问题来了：**这东西能干嘛？**

项目给了两个落地方案：

### 1. Kode Agent CLI

```sh
npm i -g @shareai-lab/kode
```

这是个开箱即用的 Coding Agent CLI，支持 Skill 和 LSP，可以接 GLM、MiniMax、DeepSeek 等开放模型。

### 2. Kode Agent SDK

这是个嵌入式 SDK，可以把 Agent 能力塞进你的应用里。和官方 Claude Code SDK 不同，它没有 per-user 进程开销，适合高并发场景。

## 八、一个更激进的想法：从"用完即走"到"始终在线"

这个项目最让我兴奋的，是它提到的**姊妹项目 [claw0](https://github.com/shareAI-lab/claw0)**。

learn-claude-code 教的 Agent 是**"用完即走"**型：开终端、给任务、做完关掉，下次重开是全新会话。

但 claw0 证明了另一种可能：**主动式常驻 AI 助手**。

它加了两个关键机制：

1. **心跳（Heartbeat）**：每 30 秒系统给 Agent 发条消息，检查有没有事可做。没事就继续睡，有事立刻行动。
2. **定时任务（Cron）**：Agent 可以给自己安排未来要做的事，到点自动执行。

再加上 IM 多通道路由（WhatsApp、Telegram、Discord 等 13+ 平台）、不清空的上下文记忆、Soul 人格系统...

**Agent 就从一个临时工具，变成了始终在线的个人 AI 助手。**

想象一下：你有个 AI 助手，它知道你所有的项目进度，会主动提醒你"那个 bug 还没修"，会在每天早上给你发今日计划，会在你睡觉时自动跑测试...

这不是科幻，这是已经实现的东西。

## 九、最后说点人话

写这篇文章的时候，我一直在想：**怎么让这件事跟你有关？**

你可能不是开发者，不想手搓 Agent。但我想告诉你的是：

**Agent 的本质，不是"更聪明的 AI"，而是"更好的工具使用者"。**

LLM 本身只是个语言模型，它不会写代码、不会运行命令、不会操作你的电脑。但当你给它加上工具、计划、记忆、团队协作...

**它就从"会说话的百科全书"，变成了"能干事的队友"。**

这个转变的关键，不在于模型本身，而在于我们如何设计机制。

就像那句格言说的：

> **模型就是智能体。我们的工作就是给它工具，然后让开。**
