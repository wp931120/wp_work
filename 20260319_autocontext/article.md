你的 Agent 很聪明。但每次上下文窗口填满，它就忘了二十分钟前琢磨出来的东西。

你重启，它犯同样的错。你修补提示词，撑了一周，新的失败模式又冒出来。凌晨两点翻日志成了常态。

这不是模型的问题。是基础设施的问题。Agent 没有跨运行的持久记忆，没有机制把"什么管用、什么不管用"带进下一次运行。每次启动都是冷启动。

autocontext 想填补的就是这个缺口。

## 一个闭环，六个角色

autocontext 的思路很直接：让 Agent 反复执行任务，每次跑完后由一组 Agent 团队分析结果，更新共享知识库。下一次运行继承前面所有的积累。

具体来说，六个专门角色轮番上阵：

**Competitor（竞争者）**：生成针对任务的策略

**Translator（翻译器）**：把原始输出转换为结构化 JSON

**Analyst（分析师）**：复盘刚才发生了什么，找根因，提建议

**Coach（教练）**：把分析提炼成战术手册的更新条目

**Architect（架构师）**：发现需要自动化的分析模式，生成工具

**Curator（策展人）**：把关知识库更新的质量，合并重复的教训

外加一个 **Orchestrator（编排器）** 负责调度、重试、并行执行。它不产生内容，但没有它什么都跑不起来。

核心机制很简单：知识只有在被验证后才会持久化。策略没改进就回滚。这不是提示词工程，是带持久化的闭环优化系统。

## 两种场景，两种验证方式

autocontext 支持两类任务，验证逻辑不同：

**游戏场景**：有明确的胜负条件。夺旗赛、资源分配、多 Agent 协调问题。好处是评分函数客观，不依赖 LLM 评审的主观判断。Elo 分数上去就是真的变强了，没有歧义。

**Agent 任务**：开放式问题。写事故复盘报告、设计临床试验方案、构造数学证明、起草网络安全响应计划。这类任务用 LLM 评审 + 多维度评分标准。

两种场景互补。游戏场景给你"假不了"的硬保证——如果知识积累有效，分数会在真实对手面前上涨。Agent 任务给你领域灵活性——同一个循环可以指向任何你能描述清楚的问题。

## 知识积累长什么样

作者用一个简单的网格夺旗场景做了测试，接 Anthropic 真实 API。两代运行积累了 33 条独立教训。策略从通用的参数扫描进化到分等级的资源密度阈值。战术手册从空白涨到 5870 字符。

这些知识会迁移。一个带着已填充战术手册启动的新 Agent，比冷启动的跑得更好。这就是复利效应。

Context Compaction（上下文压缩）是敌人。autocontext 是解药。

## 评审系统：整个闭环的命门

这个系统的质量上限取决于评审（Judge）的质量。评审烂，循环就是空转。评审好，才能越跑越强。

评审子系统的工程投入占了大部分：

**多维度评分标准**：每个任务定义具体的维度（准确性、清晰度、可操作性等）。每个维度独立打分。这意味着改进可以针对最弱的维度精准下手，而不是盲目重写一切。

**解析健壮性**：LLM 输出很乱。评审有四层降级解析器：标记分隔的 JSON → 原始 JSON 提取 → 正则模式 → 结构化重试。在对抗测试中，故意喂给系统格式错误的评审输出，每次都能干净恢复。

**对抗性验证**：用模糊评分标准、矛盾标准（"极度技术化" AND "五岁小孩能懂"）、幻觉倾向任务测试系统。最令人印象深刻的是幻觉测试：编造的主张在所有维度都得 0 分，修订版本正确指出了源材料可能不存在。

## 从前沿模型到本地模型

一个更有野心的功能：autocontext 可以把前沿模型的策略蒸馏到本地模型。

工作流程：

1. 用 Claude Opus 4.6 或 GPT-5.4 做发现阶段
2. 从运行数据库导出训练数据
3. 在 Apple Silicon 上用 MLX 训练小型本地模型
4. 当本地模型够强时，把未来的竞争者运行路由给本地模型
5. 本地模型弱时回退到前沿模型

为什么选 Apple Silicon？很多在 OpenClaw 和 Claude Code 生态里开发 Agent 的人都用 M 系列芯片。MLX 原生支持 Metal 加速，不需要 CUDA、不需要租云 GPU、不需要折腾 Docker。有台内存足够的 MacBook Pro 或 Mac Studio，本地推理的硬件就有了。

完整流水线已经实现：导出、基于 git 的实验状态机训练循环、带 GQA 注意力和 RoPE 的 MLX 模型、safetensors 检查点、以及插入同一生成循环的 MLXProvider。

## 不仅仅是写代码

场景系统设计成领域无关的。作者内部测试的场景远超软件工程：

- 形式化证明：构造和验证数学论证，每轮加强逻辑严密性
- 临床试验设计：药物试验协议生成，含随机化、人口统计学、监管约束
- 地缘政治兵棋推演：危机响应规划，多方利益相关者目标冲突，升级动态
- 金融投资组合优化：感知体制的配置策略，适应宏观经济指标
- 网络安全事件响应：分流、遏制、修复规划，对抗演进中的威胁场景
- 药物相互作用分析：评估化合物对的安全性和有效性
- 生态系统模拟：多周期环境，Agent 必须适应变化条件

模式在每种情况下都一样：运行、评估、学习、持久化、重复。Harness 不关心 Agent 在写代码还是设计临床试验协议。它只关心输出是否变好了。

## 打开黑盒看一眼

安装和运行：

```bash
git clone https://github.com/greyhaven-ai/autocontext.git
cd autocontext/autocontext
uv venv && source .venv/bin/activate && uv sync --group dev

# 跑确定性测试（不需要 API Key）
AUTOCONTEXT_AGENT_PROVIDER=deterministic uv run autoctx run \
  --scenario grid_ctf --gens 3 --run-id quickstart

# 用 Anthropic 跑
AUTOCONTEXT_AGENT_PROVIDER=anthropic \
AUTOCONTEXT_ANTHROPIC_API_KEY=your-key \
uv run autoctx run --scenario grid_ctf --gens 3
```

项目提供 CLI、实时 Dashboard、MCP Server（可插入任何 MCP 兼容的 Agent，如 Claude Code、Cursor、OpenClaw）。有 Python 包和 TypeScript 包两种选择，共享 SQLite 存储。

## 为什么开源，为什么现在

项目本来是内部工具。评估基础设施是每个项目的瓶颈。每个 Agent 部署最终都需要某种"搞清楚问题，让它变好，全力以赴，别犯低级错误"的无尽人类反馈循环。

Karpathy 的 autoresearch 工作和社区围绕递归自我改进的热情，让作者觉得时机到了。知识积累、Harness 驱动优化、前沿到本地蒸馏这些想法正在快速收敛。很多人在独立构建这些碎片。

既然社区在攻克同样的问题，闭着不放感觉不对。

---

**我的判断**：

这是 AI Agent 基础设施演进的重要一环。上下文窗口再大也有上限，模型再强也经不起每次冷启动。让 Agent 拥有"持久记忆"，能跨运行积累经验，是解决"金鱼脑"问题的正确方向。

评审系统是整个设计的瓶颈。评审质量直接决定进化方向对不对。评审烂，闭环就是空转甚至负优化。这一点作者在工程上投入了大量精力，四层降级解析、多维度评分、对抗性测试，说明他们知道这里的份量。

知识蒸馏是实用功能。前沿模型跑一轮发现阶段很贵，把学到的策略压进本地模型，后续运行可以便宜很多。边缘设备推理的趋势下，这条路有价值。
