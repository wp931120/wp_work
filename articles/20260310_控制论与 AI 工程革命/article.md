# 从蒸汽机到 AI Agent：控制论正在重塑工程师的未来

> 你不再亲手转动阀门，你学会掌舵。

---

## 01 一个似曾相识的模式

最近读到 OpenAI 的 **Harness Engineering** 文章，我总觉得这个模式在哪见过。

不是见过一次，是三次。

**第一次是 1780 年代，詹姆斯·瓦特的离心调速器。**

在此之前，工人站在蒸汽机旁边，手动调节阀门。在此之后，一个加权飞球机构自动感知转速并调节阀门。

工人没有消失。**工作变了**：从转动阀门变成设计调速器。

**第二次是 Kubernetes。**

你声明期望的状态——三个副本、这个镜像、这些资源限制。控制器持续观察实际状态。当两者偏离时，控制器自动修复：重启崩溃的 Pod、扩展副本、回滚糟糕的部署。

工程师的工作从重启服务变成了**编写系统用来校准的规范**。

**第三次就是现在。**

OpenAI 描述了一群不再亲手写代码的工程师。他们设计环境、构建反馈循环、编码架构约束——然后 Agent 来写代码。

**五个月一百万行代码，零行手写。** 他们称之为"Harness Engineering"（驾驭工程）。

每次都是同样的模式。

诺伯特·维纳在 1948 年为它命名：**控制论（Cybernetics）**，源自希腊语 κυβερνήτης（舵手）。

**你不再转动阀门。你掌舵。**

---

## 02 为什么代码库曾是最后的堡垒

每次这个模式出现，都是因为有人在某个层级构建了足够强大的**传感器**和**执行器**来关闭反馈回路。

代码库曾经有反馈回路，但只在底层。

- 编译器关闭语法层面的回路
- 测试套件关闭行为层面的回路
- Linter 关闭代码风格层面的回路

这些都是真实的控制论控制——但它们只作用于可以机械检查的属性。

**它能编译吗？它能通过测试吗？它符合规范吗？**

在这之上的所有东西——这个改动是否符合系统架构？这是正确的方法吗？这个抽象会在代码库增长时引发问题吗？——**没有传感器，也没有执行器**。

只有人类能在这个层级操作，既判断质量又编写修复。

**LLM 一次性改变了这两点。**

它们能在人类曾经拥有的层级感知——也能在同一层级行动：重构模块、重新设计不一致的接口、围绕真正重要的契约重写测试套件。

**第一次，反馈回路可以在重要决策发生的层级关闭。**

## 03 校准传感器和执行器

让基本反馈回路工作起来——Agent 能运行的测试、提供可解析输出的 CI、指向修复的错误信息——这些只是**入场券**。

Carlini 在让 16 个并行 Agent 构建 C 编译器时证明了这一点：提示词简单到令人尴尬，但**精心设计了测试基础设施**。

> "我的大部分精力都花在为 Claude 设计环境——测试、环境、反馈。"

**更难的问题是：用特定于你系统的知识来校准传感器和执行器。**

这正是大多数人卡住的地方，也是他们责怪 Agent 的地方。

> "它总是做错事。它不理解我们的代码库。"

这个诊断几乎总是错的。

**Agent 失败不是因为它缺乏能力。它失败是因为它需要的知识——对你的系统而言什么是"好"、你的架构奖励哪些模式、避免哪些模式——被锁在你的脑子里，而你还没有把它外部化。**

**Agent 不会通过渗透学习。如果你不写下来，Agent 在第 100 次运行时犯的错误和第 1 次一模一样。**

**真正的工作是：让你的判断变成机器可读的。**

- 描述真实分层和依赖方向的架构文档
- 内嵌修复说明的自定义 Linter
- 编码你团队品味的黄金原则

OpenAI 发现了完全相同的事情：他们每周五花 20% 的时间清理"AI 垃圾"——直到他们把标准编码进 Harness 本身。

## 04 唯一的出路

文档、自动化测试、编码架构决策、快速反馈回路——这些实践**一直都是正确的**。

过去三十年写的每一本工程书籍都推荐它们。

大多数人跳过它们，因为跳过的代价是缓慢且分散的：逐渐的质量下降、痛苦的入职、技术债在安静中复利。

**Agent 工程让代价变得极端。**

跳过文档，Agent 就忽略你的约定——不是在一个 PR 上，而是在每个 PR 上，以机器速度，全天候。

跳过测试，反馈回路根本无法关闭。

跳过架构约束，漂移的复利速度超过你能修复的速度。

**这就是陷阱：你不能使用 Agent 来清理烂摊子，如果 Agent 不知道"干净"长什么样。**

没有校准，制造问题的机器也无法解决问题。

**实践没有变。忽视它们的代价变得无法承受。**

## 05 生成 - 验证的不对称性

生成 - 验证的不对称性——P vs NP 背后的直觉，被 Cobbe 等人在 LLM 上实证演示——指出了方向。

**生成正确答案比验证答案更难。**

你不需要比机器更会实现。你需要比它更会评估：

- 指定什么是"正确"
- 识别输出何时偏离
- 判断方向是否正确

设计瓦特调速器的工人没有回去转动阀门。

不是因为他们不能。**因为那不再有意义。**

## 06 工程师的未来

控制论的浪潮正在席卷软件工程。

这不是第一次，也不会是最后一次。但这一次不同——**反馈回路终于关闭在了最有价值的层级**。

这意味着什么？

**意味着工程师的角色正在从"动手做"转向"动嘴说"。**

你不是在写代码，你是在告诉 Agent 什么是好代码。你不是在修复 bug，你是在定义什么是 bug。你不是在重构模块，你是在教 Agent 如何重构。

**你的判断力、你的品味、你的架构直觉——这些曾经只存在于你脑海中的东西，现在必须变成机器可读的规范。**

这不是降级。这是升级。

**从工匠到设计师。从执行者到指挥官。从转动阀门到掌舵。**

## 07 给你的行动建议

如果你想在 AI 时代保持竞争力，现在就开始：

### 1. 写下你的架构原则

别只在脑子里想。写下来。变成文档。变成机器可读的规则。

### 2. 投资测试基础设施

让 Agent 能运行你的测试。让反馈回路能快速关闭。

### 3. 构建自定义 Linter

把你的代码品味编码成自动检查。让机器知道什么是"好"。

### 4. 练习"规格驱动"思维

别急着动手写。先想清楚：什么是期望状态？如何验证？

### 5. 接受"掌舵者"身份

你不再是那个转动阀门的人。你是设计调速器的人。

## 写在最后

维纳的控制论诞生于 1948 年。

77 年后，它正在重塑软件工程。

**历史不会重复，但它押韵。**

蒸汽机工人没有消失，他们变成了调速器设计师。

传统运维工程师没有消失，他们变成了 SRE 和平台工程师。

**写代码的工程师也不会消失——但只转动阀门的人，会被设计调速器的人取代。**

AI 时代的知识管理，正在从文档库转向可执行的技能。

**你准备好了吗？**
