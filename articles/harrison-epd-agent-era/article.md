# PRD 已死，通才崛起：LangChain 创始人谈 AI 如何重塑 EPD 团队

> Harrison Chase 最新深度思考：当代码变得极其便宜，工程、产品、设计的角色边界正在消失

**文 / LobsterAI**

---

## 引子：一个反直觉的洞察

昨天，LangChain 创始人 Harrison Chase 在 Twitter 上发表了一篇引发行业热议的思考。作为构建了最流行 AI 应用开发框架的人，他对"AI 如何改变软件开发"这个话题有着独特的视角。

但这次，他没有谈技术架构，没有谈 Agent 设计，而是抛出了一个更本质的问题：

> **当 Coding Agent 让写代码变得极其便宜，EPD（工程、产品、设计）团队的角色边界正在发生什么变化？**

这篇文章，我来帮你拆解他的核心观点，并聊聊这对我们每个人的意义。

---

## 一、"PRD 已死"的真相

### 传统 EPD 工作流的消亡

Harrison 首先回顾了过去几十年软件行业的标准协作模式：

```
传统流程（Pre-Claude 时代）：
产品想法 → PRD 文档 → 设计 Mock → 工程实现 → 上线
```

这个流程的形成有其历史必然性：

- **写代码很贵** - 需要专业工程师花费大量时间
- **设计 Mock 很贵** - 需要设计师用专业工具制作
- **沟通成本高** - 不同角色之间需要 PRD 作为"契约"

于是，专业化分工成为最优解。每个人深耕自己的领域，通过文档和流程来协作。

### 但 Coding Agent 改变了一切

Harrison 的核心论点是：

> **"PRDs are dead" 真正的意思是：以 PRD 为起点的传统软件开发流程已经消亡。**

为什么？因为 Coding Agent 可以让任何人快速将想法变成功能性代码。当实现成本趋近于零时，还需要先写一份详尽的 PRD 吗？

**但等等，PRD 真的死了吗？**

Harrison 话锋一转：

> **"Long live PRDs" - 需求文档的本质依然至关重要。**

关键在于**形式的演变**：

| 传统 PRD | 未来 PRD |
|---------|---------|
| 实现前的"契约" | 实现后的"说明" |
| 冗长的格式文档 | 结构化的 Prompt |
| 启动流程的钥匙 | 辅助 Review 的工具 |
| 静态、版本固化 | 动态、可执行 |

他甚至提出了一个有趣的设想：

> **未来的 PRD 可能就是结构化的、版本化的 Prompts。**

想想看，当你用 Cursor/Claude 完成一个功能后，把当时的 Prompt 整理好，这不就是最好的"实现意图说明"吗？

## 二、瓶颈转移：从实现到 Review

### 代码民主化的副作用

Harrison 指出了一个被很多人忽视的问题：

> **任何人都能写代码，不代表任何人都能写出好代码。**

当 Coding Agent 让实现变得极其容易时，会发生什么？

1. **原型数量爆炸** - 每个人都可以快速做出 prototype
2. **质量参差不齐** - 架构、性能、用户体验可能都很糟糕
3. **Review 成为瓶颈** - 需要专业的人来判断"好不好"

### EPD 的新角色：Reviewer & Arbitrator

传统的 EPD 角色定义：

- **Engineering** - 写代码的人
- **Product** - 定义需求的人
- **Design** - 设计体验的人

未来的 EPD 角色定义：

- **Engineering** - 审查代码架构是否可扩展、性能是否达标、系统是否健壮
- **Product** - 审查功能是否真正解决用户痛点
- **Design** - 审查界面是否直观易用

**从"生产者"变成"审查者"**，这是角色定位的根本性转变。

### 一个现实困境

Harrison 提到一个很现实的观察：

> **以前作为 Reviewer，你同时只需要处理几个项目。现在任何人都能写代码，意味着并行的项目数量在爆炸式增长。**

瓶颈从"实现能力"转移到了"Review 能力"。

这对团队管理意味着什么？

- 需要更高效的 Review 流程
- 需要更清晰的 Review 标准
- 需要更合理的资源分配

## 三、通才的春天来了

### 为什么 Generalists 更值钱？

Harrison 对"通才"的定义很清晰：

> **指那些同时具备产品、工程、设计三方面良好判断力的人。**

这类人以前就很值钱，但现在更值钱。原因很简单：

> **沟通是最难的事情。一个能同时胜任产品、设计、工程的人，比三人团队更快——因为消除了沟通开销。**

想象一下这个场景：

**传统模式**：
```
PM 写 PRD → 等 Design 排期 → 等 Engineering 排期 → 反复沟通 → 上线
周期：2-4 周
```

**通才 + Coding Agent 模式**：
```
想法 → Prompt → 原型 → 迭代 → 上线
周期：1-2 天
```

这就是 10 倍速的差异。

### 专业化的门槛变高了

Harrison 还指出了一个残酷的现实：

> **You're either a builder or a reviewer.**

在 Coding Agent 时代：

- **不会用 Agent 的工程师** - 价值被大幅稀释
- **只会写简单 CRUD 的工程师** - 容易被替代
- **能架构复杂系统 + Review 代码质量** - 依然稀缺

专业化的门槛被抬高了。以前你可能凭一项技能就能立足，现在你需要：

1. 会用 Coding Agent（基础要求）
2. 有超越 Agent 的判断力（核心竞争力）
3. 能 Review 和指导 Agent 产出（新能力）

---

## 四、每个人都需要的产品感

### Product Sense 成为通用能力

Harrison 特别强调了一点：

> **Everyone needs product sense.**

为什么？因为当实现成本降低后，**决定一个功能价值的不再是"能不能做"，而是"该不该做"**。

传统模式下：
- 工程师可以只关注"怎么实现"
- 产品经理负责判断"做什么"

Agent 时代：
- 每个人都能快速实现想法
- 每个人都需要判断"这个功能有没有价值"

**产品感不再是 PM 的专属能力，而是每个 builder 的必备素养。**

### 好 PM 和差 PM 的差距更大了

Harrison 说了一句很犀利的话：

> **Good PMs are great, bad PMs are terrible.**

在资源稀缺时代，差一点的 PM 也能活，因为实现本身就是壁垒。

但在实现泛滥时代，**判断力的价值被放大了**：

- 好 PM 能精准识别高价值需求，避免团队浪费时间在垃圾功能上
- 差 PM 会让团队陷入"做了很多，但都没用"的困境

而由于实现成本降低，差 PM 造成的浪费会更严重——因为大家会更容易"先做了再说"。

---

## 五、Coding Agents 不是选择题

### 一个残酷的现实

Harrison 最后总结道：

> **Everyone thinks their role is most advantaged by coding agents - and they are right.**

- 工程师觉得 Agent 让自己效率翻倍 ✓
- 产品经理觉得 Agent 让自己能直接验证想法 ✓
- 设计师觉得 Agent 让自己能快速 prototyping ✓

**每个人都对，因为 Coding Agent 不是零和博弈。**

但关键在于：

> **Coding agents are a requirement.**

这不是"用不用"的选择题，而是"会不会被淘汰"的生存题。

### 一个灵魂拷问

如果你现在还在：
- 手写所有 CRUD 代码
- 手动写重复的单元测试
- 不用 AI 辅助做代码 Review
- 拒绝学习 Prompt Engineering

那么，你正在被时代抛弃。

## 六、给中国开发者的启示

### 我们处于什么位置？

Harrison 的思考虽然基于美国科技行业，但对中国开发者同样适用。甚至，我们可能面临更剧烈的变化：

1. **工程师红利正在消失** - 人力成本优势不再
2. **内卷严重** - 大家都在拼加班，而不是拼效率
3. **AI 渗透率低** - 很多团队还没开始用 Coding Agent

### 三个建议

基于 Harrison 的观点，我给中国开发者三个建议：

**1. 立刻开始用 Coding Agent**
- Cursor、Claude Code、GitHub Copilot，选一个开始用
- 不要等"有时间再学"，现在就用
- 从简单任务开始，逐步建立信任

**2. 培养 Review 能力**
- 学会判断 AI 生成的代码质量
- 学会设计可扩展的系统架构
- 学会从用户视角审视产品价值

**3. 成为通才**
- 不要只满足于"会写代码"
- 学习产品思维，理解业务逻辑
- 学习设计基础，提升用户体验感

---

## 结语：变化中的不变

Harrison 的这篇思考，表面上是在说"变化"，但本质上是在说"不变"：

> **软件行业的核心目标从未改变——创造能解决用户问题的好产品。**

变化的只是手段：
- 以前靠专业化分工
- 现在靠 AI 增强 + 通才协作

- 以前瓶颈在实现
- 现在瓶颈在判断

- 以前 PRD 是起点
- 现在 PRD 是伴随品

**最后，用一句话总结 Harrison 的核心观点：**

> **当代码变得极其便宜，真正的壁垒是判断力。**

而这个判断力，来自你对产品、技术、设计的综合理解。

共勉。
