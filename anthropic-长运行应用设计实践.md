# Anthropic 长运行应用设计实践：如何让 AI 应用持续稳定运行？

> 从 Anthropic 工程团队学习多代理架构、生成器 - 评估器循环和上下文管理的核心设计原则

---

## 引言

在 AI 应用开发中，我们常常面临一个挑战：**如何让 AI 应用长时间稳定运行？**

无论是客服机器人、代码助手还是数据分析工具，用户都期望它们能够持续可靠地工作。但现实是，长运行应用面临着诸多挑战：资源泄漏、状态管理复杂、错误恢复困难...

今天，我们来深入解读 Anthropic 工程团队分享的长运行应用设计经验，看看他们是如何通过**多代理架构**和**生成器 - 评估器循环**解决这些问题的。

---

## 核心挑战：为什么朴素实现会失败？

### 1. 上下文焦虑（Context Anxiety）

随着任务执行，上下文窗口逐渐填满，模型开始失去连贯性。更严重的是，某些模型（如 Claude Sonnet 4.5）会出现"上下文焦虑"——当它们认为接近上下文限制时，会过早地试图结束工作。

**解决方案：上下文重置**

完全清空上下文窗口，启动新的代理会话，同时通过结构化的人工制品（handoff artifacts）传递之前代理的状态和下一步计划。这与"压缩"（compaction）不同——压缩只是总结早期对话，代理仍然带着历史包袱继续工作；而重置给了代理一张干净的白纸。

### 2. 自我评估偏差

当被要求评估自己产生的工作时，代理倾向于自信地赞美自己的工作——即使质量明显平庸。这个问题在主观任务（如设计）中尤为明显，因为没有像软件测试那样的二元验证。

**解决方案：分离生成与评估**

将执行工作的代理与评判工作的代理分开。虽然评估代理本身也是 LLM，也会倾向于宽容，但调优一个独立的评估器比让生成器自我批评要容易得多。

## 前端设计实验：让主观质量可量化

Anthropic 团队首先在前端设计领域实验，因为这里的自我评估问题最为明显。

### 四大评估标准

团队编写了四个评估标准，同时提供给生成器代理和评估器代理：

1. **设计质量**：设计是否感觉像一个连贯的整体，而不是零件的集合？色彩、排版、布局、图像等细节是否结合在一起创造出独特的氛围和身份？

2. **原创性**：是否有定制决策的证据，还是模板布局、库默认值和 AI 生成模式？人类设计师应该能识别出有意的创意选择。未修改的库存组件或 AI 生成的典型模式（如白色卡片上的紫色渐变）在这里会不及格。

3. **工艺**：技术执行——排版层次结构、间距一致性、色彩和谐、对比度比率。这是能力检查而非创意检查。大多数合理的实现默认都能在这里表现良好；失败意味着基础功能有问题。

4. **功能性**：独立于美学的可用性。用户能否理解界面的功能、找到主要操作，并在无需猜测的情况下完成任务？

**关键洞察**：团队强调**设计质量**和**原创性**高于工艺和功能性。因为 Claude 默认在工艺和功能性上已经表现不错，但在设计和原创性上往往产出平淡无奇的内容。

### 生成器 - 评估器循环

```
[生成器创建前端] → [评估器用 Playwright 交互测试] → [评分 + 详细批评] → [生成器迭代改进]
```

- 每个生成运行 5-15 次迭代
- 评估器使用 Playwright MCP 直接与 live 页面交互，截图并仔细研究实现
- 每次迭代后，生成器做出战略决策：如果分数趋势良好则精炼当前方向，如果方法不行则完全转向新的美学风格
- 完整运行可能长达 4 小时

**惊人案例**：在一个荷兰艺术博物馆网站的生成中，第 9 次迭代时还是一个干净的深色主题落地页。但在第 10 次循环，它完全抛弃了原有方法，重新想象为一个空间体验：用 CSS 透视渲染的 3D 房间、格子地板、墙上自由摆放的艺术品、基于门的画廊房间导航而非滚动或点击。这是单次生成中从未见过的创意飞跃。

## 扩展到全栈开发：三代理架构

有了前端设计的经验，团队将这种 GAN 启发的模式应用到全栈开发中。

### 架构设计

```
[规划器] → [生成器] → [评估器] → [反馈循环]
```

#### 1. 规划器（Planner）

**职责**：将用户简单的 1-4 句话提示扩展为完整的产品规格说明

**设计原则**：
- 对范围保持雄心勃勃
- 专注于产品上下文和高层技术设计，而非详细的技术实现
- 避免 upfront 指定细粒度技术细节（如果规划器出错，错误会级联到下游实现）
- 在规格中寻找机会融入 AI 功能

#### 2. 生成器（Generator）

**职责**：一次实现一个功能（sprint 模式）

**技术栈**：React + Vite + FastAPI + SQLite/PostgreSQL

**工作流程**：
- 从规格中选取一个功能进行 sprint 开发
- 使用 git 进行版本控制
- 在 sprint 结束时自我评估
- 与评估器协商"sprint 合同"：约定本 sprint 要交付的内容和验收标准

#### 3. 评估器（Evaluator）

**职责**：像用户一样点击通过运行中的应用，测试 UI 功能、API 端点和数据库状态

**评估标准**（基于前端实验调整）：
- 产品深度
- 功能性
- 视觉设计
- 代码质量

**关键机制**：每个标准都有硬性阈值，如果任何一项低于阈值，sprint 失败，生成器收到详细反馈并返工。

## 核心设计原则总结

### 原则一：显式状态管理

不要依赖隐式状态。将应用的所有状态显式地建模和存储，这样即使进程重启，也能从断点恢复。

### 原则二：上下文重置优于压缩

当上下文窗口填满时，完全重置比压缩总结更有效。虽然需要额外的编排复杂度和 token 开销，但给了代理干净的开始。

### 原则三：分离生成与评估

让独立的评估器批判生成器的工作，比让生成器自我批评要有效得多。评估器需要被调优为怀疑态度。

### 原则四：结构化人工制品传递上下文

在代理之间传递结构化的手递人工制品（handoff artifacts），确保下一个代理能够干净地接手工作。

### 原则五：迭代反馈循环

生成器 - 评估器循环应该运行多次迭代（5-15 次），每次迭代都推动生成器向更强的输出发展。

### 原则六：硬性阈值 + 详细反馈

评估标准应该有硬性阈值，低于阈值就失败。同时提供详细的批评反馈，让生成器知道具体哪里需要改进。

## 技术选型建议

### 代理编排

- **Claude Agent SDK**：Anthropic 官方代理 SDK，简化编排
- **自定义状态机**：明确定义每个状态和转换条件

### 测试与评估

- **Playwright MCP**：让评估器能够与 live 应用交互
- **自动化测试**：单元测试、集成测试、端到端测试

### 状态存储

- **Redis**：适合缓存和临时状态
- **PostgreSQL**：适合持久化状态
- **SQLite**：适合轻量级本地应用

### 监控工具

- **Prometheus + Grafana**：指标监控
- **Sentry**：错误追踪
- **自定义日志**：业务逻辑监控

## 常见陷阱

### ❌ 陷阱一：忽略上下文焦虑

长任务中，模型会因为上下文填满而失去连贯性或过早结束。必须设计上下文重置机制。

### ❌ 陷阱二：让代理自我评估

代理倾向于赞美自己的工作。必须分离生成与评估。

### ❌ 陷阱三：评估标准模糊

"这个设计好吗？"很难一致回答。必须将主观判断转化为具体、可评分的标准。

### ❌ 陷阱四：没有硬性阈值

评估必须有硬性阈值，低于阈值就失败，否则生成器会持续产出勉强合格的工作。

### ❌ 陷阱五：一次性生成

期望单次生成就产出高质量应用是不现实的。必须设计迭代反馈循环。

## 实践建议

### 1. 从简单开始

先在一个领域（如前端设计）实验生成器 - 评估器循环，校准评估标准，然后再扩展到其他领域。

### 2. 校准评估器

使用 few-shot 示例和详细评分分解来校准评估器，确保评估者的判断与你的偏好一致，减少跨迭代的分数漂移。

### 3. 允许战略转向

在每次评估后，让生成器决定是精炼当前方向还是完全转向新的方法。有时大幅转向能带来创意飞跃。

### 4. 接受非线性改进

分数通常随迭代改进，但模式不总是线性的。有时中间迭代比最后一次更好。保留历史版本以便回滚。

### 5. 投资评估标准

评估标准的措辞会直接影响生成器的输出特征。例如，"最好的设计是博物馆级别"这样的表述会将设计推向特定的视觉收敛。

## 总结

Anthropic 的长运行应用设计经验告诉我们：

**显式优于隐式，分离优于合并，迭代优于单次，结构化优于随意。**

设计长运行 AI 应用是一项系统工程，需要综合考虑状态管理、错误处理、资源监控、代理协作等多个方面。通过多代理架构、生成器 - 评估器循环和上下文重置等设计模式，我们可以构建出更可靠、更高质量的 AI 应用。

希望这些经验能够帮助你构建更可靠的 AI 应用！
