## 一、项目概述

### 1.1 项目定位

**WeWrite** 是一个专为 **Claude Code** 设计的 AI Skill，实现微信公众号文章从选题到发布的全流程自动化。

核心理念：**一句话触发，全流程自动执行**。

```
用户：「用 demo 的配置写一篇公众号文章」
     ↓
Agent：热点抓取 → 选题生成 → 框架选择 → 文章写作 → SEO优化 → 配图生成 → 排版推送
     ↓
结果：文章已推送至微信草稿箱，等待人工审核发布
```

### 1.2 与传统方案的差异

| 维度 | 传统方案 | WeWrite |
|-----|---------|---------|
| **触发方式** | 多步骤手动操作 | 一句话触发全流程 |
| **热点来源** | 单一平台 | 微博+头条+百度多平台聚合 |
| **SEO 评估** | 纯 LLM 猜测 | 调用百度/360 API 数据化评分 |
| **风格适配** | 通用模板 | 客户级 Playbook 个性化 |
| **学习机制** | 无 | 从人工修改中提取风格规律 |
| **排版发布** | 手动复制粘贴 | CLI 自动推送到草稿箱 |

## 三、核心模块详解

### 3.1 热点抓取模块（fetch_hotspots.py）

**功能**：聚合微博、头条、百度三大平台实时热搜

**输出格式**：
```json
[
  {
    "title": "热搜标题",
    "source": "weibo",
    "hot": 1234567,
    "url": "https://...",
    "description": "热点描述"
  }
]
```

**设计亮点**：
- 多平台聚合，避免单一平台偏见
- 返回标准化 JSON，便于 LLM 处理
- 降级方案：脚本失败时回退到 WebSearch

### 3.2 SEO 关键词分析模块（seo_keywords.py）

**功能**：调用百度、360 搜索建议 API，量化关键词 SEO 价值

**输入**：关键词列表（如 `AI大模型`, `科技股`）

**输出**：
```json
{
  "keyword": "AI大模型",
  "seo_score": 8,
  "related_keywords": ["大模型应用", "AI创业", "LLM"]
}
```

**设计亮点**：
- 数据化 SEO 评估，避免纯 LLM 猜测
- 提供相关关键词，辅助选题扩展
- 评分范围 0-10，便于纳入选题综合评分

### 3.3 选题评估系统（topic-selection.md）

**评估维度**：

| 维度 | 权重 | 评分依据 |
|-----|------|---------|
| **热度分** | 30% | 热搜排名、多平台覆盖 |
| **相关度分** | 40% | 与客户 topics 匹配度、blacklist 命中检测 |
| **切入价值分** | 30% | 反直觉点、争议性、展开空间 |

**综合评分公式**：
```
总分 = 热度 × 0.3 + 相关度 × 0.4 + 切入价值 × 0.3
```

**历史去重规则**：
- 最近 7 天已覆盖 → 扣 3 分，标注"近期已覆盖"
- 7-30 天内出现 → 扣 1 分，标注"月内有相关文章"
- 超过 30 天 → 不扣分

**content_style 加成**：
- 干货型账号：方法论/工具选题 +2 分
- 故事型账号：人物/情节选题 +2 分
- 情绪型账号：共鸣/痛点选题 +2 分
- 热点型账号：热搜前 10 选题 +2 分
- 测评型账号：对比/测评选题 +2 分

### 3.4 写作框架系统（frameworks.md）

**5 套框架类型**：

| 框架 | 适用场景 | 结构特点 |
|-----|---------|---------|
| **痛点型** | 解决问题类选题 | 痛点共鸣 → 放大 → 解决方案 → 实操验证 |
| **故事型** | 人物/事件类选题 | 悬念钩子 → 背景铺垫 → 转折高潮 → 深度解读 |
| **清单型** | 盘点/推荐类选题 | 价值承诺 → 清单项 1-N → 总结+彩蛋 |
| **对比型** | 选择/决策类选题 | 选择困境 → A分析 → B分析 → 对比总结 |
| **热点解读型** | 新闻/行业动态 | 事件速览 → 表面信息 → 深层分析 → 影响预判 |

**框架选择逻辑**：
```
干货型账号 → 优先痛点型、清单型
故事型账号 → 优先故事型、热点解读型
情绪型账号 → 优先故事型、痛点型
热点型账号 → 优先热点解读型、对比型
测评型账号 → 优先对比型、清单型
```

### 3.5 去 AI 痕迹系统（writing-guide.md）

**必须删除的词汇**：

| 类别 | 示例词汇 |
|-----|---------|
| **连接词** | 首先、其次、再者、最后、总之、综上所述 |
| **AI 惯用语** | 作为一个、让我们、值得注意的是、需要指出的是 |
| **空洞形容** | 非常重要、至关重要、不言而喻、发挥着重要作用 |

**必须添加的元素**：
- 口语化表达：说实话、讲真、你猜怎么着
- 个人观点标记：我觉得、我的看法是、以我的经验
- 不完美感：偶尔自嘲、偶尔用不太精确但生动的比喻

**段落节奏要求**：
- 禁止每段都是 3-4 句话的匀称结构
- 必须穿插 1 句话的短段落（语气强调、转折、吐槽）
- 长段落不超过 150 字
- 短段落和长段落交替出现

### 3.6 Markdown 转换器（converter.py）

**核心功能**：将 Markdown 转换为微信兼容的内联样式 HTML

**技术实现**：
1. **标题提取**：从 H1 提取微信标题字段，并从正文移除
2. **Markdown 解析**：使用 python-markdown + 扩展
3. **代码块增强**：添加 data-lang 属性显示语言标签
4. **图片处理**：提取引用、添加响应式样式
5. **内联样式应用**：从 YAML 主题读取样式规则
6. **微信兼容修复**：
   - 强制所有 `<p>` 标签显式设置 color
   - 确保 `<pre>` 保留空白符

**输出结构**：
```python
@dataclass
class ConvertResult:
    html: str          # 微信兼容 HTML（body 内容）
    title: str         # 提取的 H1 标题
    digest: str        # 自动生成的摘要（≤120字节）
    images: list[str]  # 图片引用列表
```

### 3.7 视觉 AI 模块（image_gen.py）

**支持的双 Provider**：

| Provider | 适用场景 | 配置来源 |
|----------|---------|---------|
| **doubao** | 中文提示词效果好，国内访问快 | 火山引擎 Ark API |
| **openai** | DALL-E 3，国际通用 | OpenAI API |

**尺寸规格**：
- 封面：2.35:1 微信封面比例
- 内文配图：16:9 横版

**配图策略**：
- 数据/场景/转折处优先配图
- 纯观点段可不配
- 总数 3-6 张
- 间隔 ≥ 300 字
- 不在开头和 CTA 处配图

### 3.8 排版主题系统

**4 套预置主题**：

| 主题 | 风格 |
|-----|------|
| professional-clean | 专业简洁（默认） |
| tech-modern | 科技风（蓝紫渐变） |
| warm-editorial | 暖色编辑风 |
| minimal | 极简黑白 |

**主题 YAML 结构**：
```yaml

  text: "#333333"
  background: "#ffffff"
  primary: "#1a73e8"

  body_font: "system-ui, sans-serif"
```

## 四、学习与进化机制

### 4.1 Playbook 系统

**Playbook 是什么**：

Playbook 是客户的个性化写作规范，优先级高于通用 writing-guide.md。

**生成方式**：
1. 用户将历史推文放入 `clients/{client}/corpus/` 目录
2. 运行 `build_playbook.py --client {client}`
3. 脚本分析语料，提取风格特征
4. 生成 `playbook.md`

**建议语料量**：至少 20 篇历史文章，50+ 篇效果更好

### 4.2 人工修改学习机制

**触发条件**：用户说"我改了，学习一下"、"学习我的修改"

**执行流程**：
1. 获取 AI 初稿和人工终稿
2. 运行 `learn_edits.py` 进行 diff 分析
3. 对修改分类：用词替换、段落删除/新增、结构调整、标题修改、语气调整
4. 写入 `lessons/` 下的 diff YAML 文件

**自动固化**：每积累 5 次 lessons，自动触发 Playbook 更新

**示例输出**：
```yaml
edits:
  - type: "用词替换"
    original: "讲真"
    revised: "坦白说"
    position: "第3段"
    
patterns:
  - pattern: "避免使用'讲真'，改用'坦白说'"
    frequency: 3
```

### 4.3 效果复盘系统

**触发条件**：用户问"文章数据怎么样"、"效果复盘"

**执行流程**：
1. 调微信数据分析 API 拉取最近 7 天文章数据
2. 匹配 history.yaml 中的文章记录
3. 回填 stats 字段（阅读量、分享量、点赞量、阅读率）

**分析维度**：
- 哪篇文章表现最好？为什么？（标题策略？选题热度？框架类型？）
- 哪篇表现不好？可能的原因？
- 对后续选题/标题/框架的调整建议

## 五、错误处理与降级策略

### 5.1 分级降级方案

| 步骤 | 失败场景 | 降级方案 |
|------|---------|---------|
| 热点抓取 | 脚本报错/返回空 | WebSearch 替代 |
| 选题生成 | 列表为空 | 请用户手动给选题 |
| SEO 分析 | API 失败 | 回退到 LLM 判断 |
| 封面生成 | 图片生成失败 | 输出提示词，用户自行生成 |
| 推送草稿 | API 失败 | 生成本地 HTML，手动操作 |
| 历史写入 | 写入失败 | 警告但不阻断流程 |
| 效果数据 | 拉取失败 | 告知可能需等 24h（微信数据延迟） |
| Playbook | 不存在 | 正常——用通用规则 |

### 5.2 交互模式 vs 自动模式

**自动模式（默认）**：
- 不中途停下来问用户选择
- 自动选最优选题、最优框架
- 只在出错时才停下来

**交互模式**：
- 触发词：用户说"交互模式"、"我要自己选"
- 在选题/框架/配图处暂停等确认

## 八、优势与局限

### 8.1 核心优势

1. **全流程自动化**：从热点到草稿箱，一句话搞定
2. **数据驱动决策**：SEO 评分、历史去重、偏好分析都基于真实数据
3. **风格个性化**：Playbook 系统实现客户级风格适配
4. **持续进化**：从人工修改中学习，越用越懂客户风格
5. **降级完备**：每步都有 fallback，流程健壮性高
6. **工具链独立**：toolkit 可脱离 Claude Code 独立使用

### 8.2 潜在局限

1. **依赖外部 API**：微信公众号 API、图片生成 API 的稳定性和费用
2. **语料门槛**：Playbook 生成需要 20+ 篇历史文章
3. **微信平台限制**：微信对 HTML 样式有严格限制，部分效果无法实现
4. **模型依赖**：核心能力依赖 Claude 的指令跟随能力
5. **中文优化**：去 AI 痕迹规则主要针对中文，其他语言适配需调整
