# Agent 的数据库困局，Deeplake 想用"沙盒实例"解决

每过半年，AI Agent 的记忆方案就变一次。每过半年，它就崩一次。

这是 Deeplake 团队在推特上抛出的观察。看似夸张，但确实戳中了一个行业痛点——当所有人都盯着模型能力飞涨时，Agent 的数据层基础设施一直停留在原地。

## "上下文墙"是怎么来的

2024 年的主流方案是向量数据库：把相关内容塞进 prompt，就叫它记忆。这招在问答场景还行，但 Agent 不只是要回答问题，它要干活。

到了 2025 年初，行业转向让模型直接写 SQL。比向量库强，但问题来了——给每个 Agent 配一个 Postgres 实例？成本扛不住。传统数据库启动要几分钟，计算存储绑在一起，闲着也要钱。更关键的是，状态没法在 Agent 之间共享。

于是大家又退回到文件系统：给 Agent 一个 bash，一个临时目录，让它写 memory.md。这方案能用，不是因为它对，是因为它简单。

现在卡在两个烂选项中间：

文件系统扩展不了。单 Agent 读写没问题，一旦有一群 Agent 并发干活、要查百万级交互日志、需要 schema 约束别让 Agent 自己瞎编数据结构——文件系统就露馅了。它本来就不是数据库。

传统数据库也不是给 Agent 用的。给每个 Agent 开一个 Postgres 听起来合理，但传统数据库没有"缩容到零"的概念。Agent 睡 6 小时，数据库照样跑着，你照样付钱。

## Deeplake 的解法

Deeplake 是 Activeloop 推出的产品，核心思路是：给每个 Agent 一个沙盒化的 Postgres 实例，用完即毁，按需付费。

几个关键设计：

**沙盒分支**

因为计算和存储分离，Deeplake 可以瞬间克隆一个数据库或表的分支。Agent拿到自己的沙盒副本，随便折腾。失败了？扔掉。成功了？合并回主分支。

这解决了 Agent 需要"安全实验空间"的问题——它可以测各种逻辑，不用担心搞坏生产数据或踩到别的 Agent。

**弹性计算**

Agent 的工作负载天生是尖刺形的。可能闲着几小时，然后突然起 50 个子 Agent 疯狂并发 30 秒。Serverless Postgres 计算层几秒内拉起，扩容处理峰值，任务结束缩回零。你不用为闲置的数据库机器付钱。

**无限存储**

数据存在 S3 而不是挂载盘（EFS），存储容量不再受本地磁盘限制。Agent 跑几天甚至几周，积累大量中间状态、日志、上下文——都能装得下。

## 多模态：传统数据库的盲区

这是最后一块拼图。企业数据不只是行列。图片、音频、视频、向量嵌入——Agent 需要一个能统一存储和查询这些东西的地方。

Deeplake 的定位是"像 Databricks Lakebase，但原生多模态、为写入而非只读副本设计"。

它把碎片化的工具需求统一了：不需要 Agent 同时折腾向量数据库（存嵌入）、S3 桶（存图片）、JSON 文件（存关系状态）。所有东西塞进同一个沙盒 Postgres，Agent 可以用 SQL 把关系数据（比如用户历史偏好）、向量相似性搜索（找相关文档）、图片引用放在一条查询里关联。

## 怎么用

```bash
pip install deeplake

# 安装 skills
npx skills add activeloopai/deeplake-skills

# 执行
claude "create a deeplake table from documents and enable lexical search."
```

拿到 DEEPLAKE_API_KEY 后，可以扔给 Claude Code 或 Codex 让它帮你建表、索引嵌入、导入 PDF 或视频。

## 一点看法

模型越来越聪明，需要的"手把手教导"越来越少。但数据持久化的物理现实不会变——模型的上下文窗口装不下整个世界。

我们给模型提供了沙盒 REPL 来安全执行无状态计算，却迟迟没给它一个沙盒数据库来安全执行有状态操作。这在逻辑上说不通。

如果数据即状态，那 Deeplake 想做的，就是给 AI 一套数据运行时。让 Agent 真正能干活，而不是只会回答问题。
