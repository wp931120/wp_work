# 0.9B 小模型干翻 235B 大佬：智谱 GLM-OCR 技术报告解读

> 模型就是智能体。我们的工作就是给它工具，然后让开。

---

## 一、先说个反直觉的结论

**模型越大越好？不一定。**

就在几天前（2026 年 3 月 11 日），智谱 AI 开源了一个只有 **0.9B 参数**的 OCR 模型 GLM-OCR。结果呢？

它在 OmniDocBench v1.5 榜单上打出了 **94.6 分**，不仅超过了 PaddleOCR-VL-1.5（94.5 分），还把 Qwen3-VL-235B（89.15 分）、Gemini-3 Pro（90.33 分）这些"巨无霸"甩在身后。

你没看错：**一个不到 10 亿参数的小模型，干翻了 2350 亿参数的行业标杆。**

这篇文章，我想带你拆解 GLM-OCR 的技术报告，看看它到底做对了什么。更重要的是，我想告诉你：**在 AI 工程化的世界里，"小"有时候比"大"更有力量。**

---

## 二、核心洞察：OCR 不是生成任务，是"抄写"任务

先抛出一个关键洞察，这是整个 GLM-OCR 设计的起点：

> **OCR 是确定性任务，不是创造性任务。**

什么意思？

当你让 GPT-4 写一首诗，它需要"创造"内容 —— 这是生成任务。但当你让它识别一张发票上的文字，答案早就写在纸上了，它只需要"抄下来" —— 这是确定性任务。

传统 LLM 的自回归解码（一个 token 一个 token 往外蹦）是为生成任务设计的，用在 OCR 上就像**用法拉利送外卖**——性能过剩，效率还低。

GLM-OCR 的第一个突破点就在这里：**它引入了 Multi-Token Prediction（MTP）机制**。

---

## 三、Multi-Token Prediction：一次预测 10 个 token

先上代码，看看 MTP 的核心思想：

```python
# 传统自回归：一次预测 1 个 token
for i in range(sequence_length):
    token = model.generate_one_token(context)
    context.append(token)

# MTP：一次预测 k 个 token
for i in range(0, sequence_length, k):
    tokens = model.generate_k_tokens(context, k=10)
    context.extend(tokens)
```

GLM-OCR 在训练时学会了一次预测 **10 个 token**，在实际推理时平均每次能生成 **5.2 个 token**。这意味着什么？

**解码步数直接减少 80%，吞吐量提升约 50%。**

更关键的是，MTP 不是简单的"批量预测"。它通过共享参数的辅助预测头，让模型学会"往前看"—— 这对于表格、公式这种结构化输出特别重要。

想象一下：你在抄写一个表格，如果一次只看一个字，很容易抄错行列；但如果一次看一整行，结构就清晰多了。MTP 就是这个道理。

## 四、两阶段流水线：先分割，再识别

GLM-OCR 的第二个设计亮点是**两阶段流水线**：

```
输入图片 → 布局分析（PP-DocLayout-V3） → 区域分割 → 并行识别 → 合并输出
```

为什么要这么麻烦？直接丢给模型不行吗？

**还真不行。**

论文里提到一个小模型常见的毛病：**幻觉和重复生成**。当文档布局复杂时（比如多栏、跨页表格），小模型很容易"看花眼"，开始胡言乱语。

通过先做布局分析，把复杂文档切成多个独立区域，每个区域单独识别，最后再按阅读顺序合并 —— 这就把一个大问题拆解成了多个小问题。

**这就像让你整理一个乱糟糟的房间：先分区（客厅、卧室、厨房），再逐个收拾，比一上来就乱捡要高效得多。**

## 五、统一框架：文档解析 + 关键信息提取

GLM-OCR 支持两类任务：

1. **文档解析**：输入图片，输出结构化 Markdown/JSON
2. **关键信息提取（KIE）**：输入图片 + 提示词，输出指定字段的 JSON

看起来是两回事，但 GLM-OCR 把它们统一成同一个问题：

> **都是"给定视觉输入，生成结构化输出"。**

区别只在于：
- 文档解析：输出完整的文档结构
- KIE：输出指定字段

这种统一设计的好处是**参数效率高**，模型学到的文档表示可以在两个任务之间迁移。

举个例子，你让它提取发票信息：

```python
# 输入提示词
prompt = """
提取以下发票的关键信息，输出 JSON 格式：
- 发票代码
- 发票号码
- 开票日期
- 金额
"""

# 输出
{
    "发票代码": "123456789012",
    "发票号码": "98765432",
    "开票日期": "2026 年 03 月 15 日",
    "金额": "¥1,234.56"
}
```

## 六、训练配方：四阶段渐进式学习

GLM-OCR 的训练分为四个阶段，每个阶段都有明确目标：

| 阶段 | 任务 | 数据规模 | 关键目标 |
|------|------|----------|----------|
| Stage 1 | 视觉编码器训练 | 数百亿图像 - 文本对 | 建立视觉表示能力 |
| Stage 2.1 | 视觉 - 语言预训练 | 文档解析 +VQA+ 定位 | 多模态对齐 |
| Stage 2.2 | 预训练 + MTP | 文档解析 +VQA+ 定位 | 适应多 token 预测 |
| Stage 3 | SFT + MTP | OCR/公式/表格/KIE | 任务特化 |
| Stage 4 | 强化学习 | 困难样本 | 提升结构化输出可靠性 |

**Stage 4 的强化学习设计尤其值得注意。**

他们为不同任务设计了专门的奖励函数：

| 任务 | 主要奖励 | 额外约束 |
|------|----------|----------|
| 文本识别 | 归一化编辑距离 | 重复惩罚 |
| 公式识别 | CDM 分数 | 结构有效性检查 |
| 表格识别 | TEDS 分数 | 标签闭合验证 |
| KIE | 字段级 F1 | JSON 解析验证 |

**这就像教学生考试：不仅要看分数，还要检查答题格式规不规范。**

## 七、性能对比：小模型的逆袭

来看硬指标。下面是 OmniDocBench v1.5 的部分结果：

| 模型 | 参数量 | 综合得分 | 文本编辑↓ | 公式 CDM↑ | 表格 TEDS↑ |
|------|--------|----------|-----------|-----------|------------|
| GLM-OCR | **0.9B** | **94.62** | 0.040 | 93.90 | **93.96** |
| PaddleOCR-VL-1.5 | 0.9B | 94.50 | **0.035** | **94.21** | 92.76 |
| MinerU2.5 | 1.2B | 90.67 | 0.047 | 88.46 | 88.22 |
| Qwen3-VL | 235B | 89.15 | 0.069 | 88.14 | 86.21 |
| Gemini-3 Pro | - | 90.33 | 0.065 | 89.18 | 88.28 |

**几个关键观察：**

1. **GLM-OCR 在表格识别上绝对领先**（93.96 TEDS），这是它的杀手锏
2. 文本识别略输 PaddleOCR-VL-1.5（0.040 vs 0.035），但差距很小
3. **用 0.9B 参数打败 235B 模型**，这不是运气，是架构设计的胜利

再看真实场景的表现：

| 任务 | GLM-OCR | PaddleOCR-VL-1.5 | Gemini-3 Pro |
|------|---------|------------------|--------------|
| 印章识别 | **90.5** | 42.2 | 91.3 |
| 多语言文本 | **69.3** | 54.8 | 86.2 |
| 代码文档 | **84.7** | 75.8 | 86.9 |
| 真实表格 | **91.5** | 86.1 | 90.6 |
| 收据 KIE | **94.5** | - | 97.3 |

**印章识别 90.5 分**这个成绩特别亮眼 —— 比 PaddleOCR-VL-1.5 高出 48 分，几乎追平 Gemini-3 Pro。

GLM-OCR 的吞吐量对比（单副本、单并发）：

| 模型 | 图像输入 (页/秒) | PDF 输入 (页/秒) |
|------|------------------|------------------|
| **GLM-OCR** | **0.67** | **1.86** |
| PaddleOCR-VL-1.5 | 0.39 | 1.22 |
| Deepseek-OCR2 | 0.32 | - |

**PDF 处理速度 1.86 页/秒**，是 PaddleOCR-VL-1.5 的 1.5 倍，MinerU2.5 的 4 倍。

更狠的是价格。GLM-OCR 的 MaaS API 定价：

> **0.2 元 / 百万 tokens**

算下来：
- **1 块钱可以处理约 2000 张 A4 扫描图片**
- 或者 200 页简单布局的 PDF（每页 10 页）

**成本是传统 OCR 方案的十分之一。**

## 九、使用方式：SDK + 基座模型双模式

GLM-OCR 提供两种使用方式：

### 1. SDK 模式（完整文档解析）

```python
from glm_ocr import GLMOCR

ocr = GLMOCR()
result = ocr.process("document.pdf")
# 输出结构化 Markdown + JSON
```

适合企业级生产环境，支持布局感知、多元素解析。

### 2. 基座模型模式（轻量任务）

通过提示词控制不同任务：

```python
# 文本识别
prompt = "Text Recognition: [image]"

# 表格识别
prompt = "Table Recognition: [image]"

# 公式识别
prompt = "Formula Recognition: [image]"

# 关键信息提取
prompt = """
Extract the following fields from this invoice in JSON format:
- invoice_code
- invoice_number
- date
- amount
"""
```

适合快速原型、特定任务场景。

## 十、局限性与未来

论文也坦诚地列出了几个局限性：

1. **两阶段架构的误差传播**：布局检测错了，后面识别也会受影响
2. **数据覆盖限制**：极低分辨率、严重扭曲的文档表现会下降
3. **结构化输出的随机性**：换行、空格处理可能有小幅波动
4. **KIE 依赖提示词质量**：提示词写不好，提取效果会打折

这些都是未来改进的方向。

## 十一、核心 takeaway

最后，我想用论文里的一句话收尾：

> **This report introduces GLM-OCR as a practical solution for structured document understanding under real-world system constraints. Instead of relying on large model scaling, the design prioritizes controllable latency, memory efficiency, and structured output reliability.**

翻译过来就是：

**GLM-OCR 不是靠"堆参数"赢的，是靠"设计"赢的。**

它做对了三件事：

1. **识别任务本质**：OCR 是确定性任务，不需要创造性生成
2. **架构创新**：MTP + 两阶段流水线，针对性优化
3. **工程导向**：部署效率、成本、可维护性，一个都不少

**这对我们有什么启发？**

如果你的业务场景也有类似的"确定性任务"（比如数据录入、格式转换、信息提取），别一上来就想着上多大模型。

**先理解任务本质，再设计合适的架构。**

有时候，**小模型 + 好设计 > 大模型 + 通用方案**。
